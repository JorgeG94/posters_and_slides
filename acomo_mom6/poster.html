Objective why standard Fortran

One source code for CPU and GPU

Keep a single source of truth that works for both CPUs and multiple GPU vendors. The ocean physics stays in Fortran for the community; the GPU parallelism is offloaded to the compiler.

do concurrent to express parallelism

A standard-language construct, not a vendor pragma. do concurrent maps the same loop to multicore CPU or to the GPU — no directives, no kernels, no rewrite. Data movement is handled with OpenMP.

Bitwise reproducible

Goal: maintain the bitwise-reproducibility contract from MOM6 — identical bits across decompositions and devices.

OpenMP for data movement

OpenMP directives are used to manage data movement between the host CPU and the device GPU, portable across compilers.

Compute runs in do concurrent; OpenMP target … map keeps A resident on the GPU, so PCIe isn’t crossed every step.
Thanks to the compiler!

do concurrent for
CPU and GPU parallelism.

CPUs and GPUs like different styles of parallelism. Blocking over the vertical layers appears to be the straight winning choice. One source, portable, readable and maintainable.
Tile the column into nblocks; each block is a single do concurrent over its levels and the full i×j plane. Block size bk is the per-device knob — large for GPU, small for CPU.

Approach GPU porting

CPUs and GPUs are not equal

GPUs desire throughput; any serialization has to come after distribution. CPUs handle divergent code paths and focus on vectorization. Blocking over the vertical layers reveals a compromise.

Same array A(i,j,k); only the k-loop moves — outer on CPU, inner on GPU.

Column-wise serialization is still a killer

Blocking unifies the horizontal physics (most of it). Vertical recurrences (mixing, tridiagonal solves) keep k sequential inside on every device — same nest, block size just pinned.

block_size is the only knob to turn

bk=nz → one big launch (GPU); bk=1 → level-by-level (CPU); in between trades launches against footprint.

Performance is memory-bound

V100 is 900 GB/s while a full Saphire Rapids is ~600 GB/s. Maximum speedup for a memory-bound workload is limited by the memory bandwidth. Max speedup on Gadi is ~6x per node.

The cost of reproducibility

GPU atomics and reductions reorder floating-point ops. Staying bit-for-bit means refusing the fast nondeterministic path and enforcing fixed-order arithmetic on the device.

Methods & systems

MOM6 · nvfortran -stdpar -O3 · NCI Gadi (Xeon Sapphire Rapids, V100, H100) · single source, CPU + GPU.
