Objective why standard Fortran
Same array A(i,j,k); only the k-loop moves — outer on CPU, inner on GPU.

One source code for CPU and GPU

Keep a single source of truth that works for both CPUs and multiple GPU vendors. The ocean physics stays in Fortran for the community; the GPU parallelism is offloaded to the compiler.

Bitwise reproducible

Goal: maintain the bitwise-reproducibility contract from MOM6: identical bits across decompositions and devices.

Compute runs in do concurrent; OpenMP target … map keeps A resident on the GPU, so the interconnect isn’t crossed every step.
Thanks to the compiler!

do concurrent for
CPU and GPU parallelism.

Tile the column into nblocks; each block is a single do concurrent over its levels and the full i×j plane. The block size bk (levels per block, of nz total) is the per-device knob — large for GPU, small for CPU.

Approach GPU porting

do concurrent to express parallelism

A standard-language construct, not a vendor pragma. do concurrent maps the same loop to multicore CPU or to the GPU .

OpenMP for data movement

Directives move data between host and device this is portable across compilers, while compute stays in plain Fortran.

CPUs and GPUs are not equal

GPUs desire throughput; any serialization has to come after distribution. CPUs handle divergent code paths and focus on vectorization. Blocking over the vertical layers reveals a compromise.

Multi-GPU via GPUDirect

Multi-GPU runs are enabled with GPU-direct communication: devices exchange halos directly, without staging through host memory.

Blocks optimize both CPU and GPU

One block size bk (levels per block) tunes the whole port: bk=nz → one big launch (GPU); bk=1 → level-by-level (CPU); in between trades launches against footprint.

Conclusions

The dynamical core of MOM6 has been successfully ported to GPUs with good performance on both CPUs and GPUs. The parametrizations and physics are underway! Expect GPU accelerated MOM6 at a cinema near you soon!
